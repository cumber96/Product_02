## Background
1차 사용자 리뷰에서 아래 문제가 보고됨:
- 기록을 추가하면 앱 푸시가 너무 많이 발송됨

  원인으로 단정하지 않고 먼저 재현 → 원인 분석 → 문제 정의 후 수정한다.

- 한 번의 저장 동작에는 의도한 알림만 필요한 횟수만큼 발송되어야 함

   기록을 1건 추가 → 푸시가 3번 발송됨.
2. "기록한 생리 일수에 비례"하는지 별도로 검증:
   구독을 1개만 남긴 상태에서 종료일까지 포함한 5일짜리 기록을 추가 →
   푸시는 여전히 1번만 발송됨.
   → 날짜 범위(생리 일수)와 발송 횟수는 무관함을 확인. 사용자의 최초 추정은 기각.

## 확인한 근본 원인
`functions/_lib/push.js`의 `notifyUser()`는 해당 사용자의
`push_subscriptions` 테이블에 저장된 **모든 row**에 각각 푸시를 보낸다.

`functions/api/push/subscribe.js`의 구독 저장 로직은 동일한 `endpoint`에
대해서만 upsert(`ON CONFLICT(endpoint) DO UPDATE`)했고, 사용자가
PWA 재설치·브라우저 재구독 등으로 새로운 endpoint를 받을 때마다
이전 endpoint row는 정리되지 않고 계속 쌓이는 구조였다.

즉, "생리 일수에 비례"처럼 보인 것은 우연이었고, 실제로는
"그 사용자에게 누적된 구독(row) 개수만큼 기록 1건당 중복 발송"되는 문제였다.

## 수정한 내용과 파일
- `functions/api/push/subscribe.js`
  - 새 구독을 저장하기 전에, 같은 `user_id`의 다른 endpoint 구독을 먼저 삭제.
  - 이 앱은 1인 1기기 사용을 전제로 하므로(Decision 004, 단순함 우선),
    사용자당 활성 구독을 항상 1개로 유지하도록 함.

## 수행한 테스트와 결과
로컬 wrangler dev + D1 local + fake push endpoint(로컬 HTTP 서버)로 실제 발송 횟수 검증.

1. (수정 전 재현) 구독 3개 상태에서 기록 추가 → 푸시 3번 발송 확인.
2. 구독 1개 상태에서 5일 범위 기록 추가 → 푸시 1번만 발송 확인 (날짜 범위 무관 확인).
3. (수정 후) 구독 3개가 남아있는 상태에서 새 endpoint로 재구독(`POST /api/push/subscribe`) →
   기존 3개 row가 삭제되고 새 endpoint 1개만 남는 것 확인.
4. 재구독 후 기록 추가 → 새 endpoint로 정확히 1번만 발송되는 것 확인.
- `node --input-type=module --check`로 `subscribe.js` 구문 검증 통과.
- 테스트에 사용한 로컬 세션/유저/구독/기록 데이터는 모두 정리함.

## 기존 기능에 미치는 영향
- 기록 추가/수정/삭제 시 알림 발송 로직 자체(`notifyPartner`, `sendWebPush`)는 변경하지 않음.
- 구독 저장 API(`POST /api/push/subscribe`)만 변경: 사용자당 최신 구독 1개만 유지.
- 정상적인 단일 기기 사용 흐름에는 영향 없음 (기존과 동일하게 동작).

## 남아 있는 위험
- **운영(remote) DB에는 이미 쌓여 있는 중복 구독 row가 남아있을 수 있음.**
  이번 수정은 "다음에 그 사용자가 다시 구독할 때" 정리되는 방식이라,
  재구독이 일어나기 전까지는 운영 DB의 기존 중복이 자동으로 사라지지 않는다.
  배포 시점에 `wrangler d1 execute --remote`로 중복 row를 한 번 정리하는 것을 권장.
  (원격 DB 접근 권한이 없어 이번 세션에서는 직접 정리하지 못함)
- 만약 향후 한 사람이 여러 기기(폰+태블릿 등)에서 동시에 알림을 받고 싶어지면,
  현재 방식(사용자당 구독 1개 강제)은 그 요구를 막는다. 지금은 1인 1기기 전제로 설계.
